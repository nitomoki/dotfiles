# 実行環境

この `~/.claude/CLAUDE.md` は dotfiles 経由で **全マシン共通**に配布される。Claude Code は下記3台のいずれでも動くため、**今どのマシンで動いているかを必ず `hostname` で判定する**こと。取り違えないこと。

## 全体構成

```
[Windows]  →  [WSL2 (Ubuntu)]  →  [Nucbox (Ubuntu, mini PC)]
 WezTerm GUI     mosh/ssh で接続       mosh/ssh で接続
```

`hostname` による判定：

- `tomoki-NucBox-G10` → Nucbox（mini PC）
- それ以外 → WSL2 か Windows ネイティブ

3 台すべてで dotfiles と Claude Code が動く。

## 含意（取り違えやすいポイント）

- 実行するシェルコマンドは**今いるマシンのローカル**で走る。Nucbox には `/mnt/c/...` のような Windows パスは無い（WSL2 には存在する）。
- WezTerm GUI 本体は Windows 上で動いている。Nucbox / WSL2 側で `~/dotfiles/.config/wezterm/` を編集しても**手元の表示には即反映されない**（dotfiles を Windows 側にデプロイして初めて効く）。
- IME（Windows IME / Mozc）も Windows 側にある。Nucbox など遠隔の Linux マシンから IME を制御したい場合は OSC エスケープシーケンス経由でローカル WezTerm に通知する構成になる（`im-select.exe` 等は Linux マシンには無い）。

# ローマ字直書き入力への対応

ユーザーは iPhone など IME が使いにくい環境から、**かな漢字変換せずローマ字のまま**メッセージを送ることがある（例：`konoyouninihongowohennkannsezunyuuryokusuru`）。以下を守ること。

## 基本方針

- ローマ字直書きと判定したら、頭の中で日本語に復元してから応答する。応答自体は通常通り日本語（漢字仮名混じり）で返す。
- **復元結果をユーザーに復唱する必要は無い**（毎回「〜という意味ですね」と返すと冗長）。ただし作業の起点になる重要な指示の場合は、最初の一回だけ確認してよい。

## 曖昧性が出たときは必ず確認する

ローマ字には同音異義語・語境界・送り仮名の曖昧性がある。**勝手に解釈して作業を進めず、ユーザーに聞き返す**こと。特に以下のパターン：

- **同音異義語**：`kami`（紙/神/髪/上）、`hashi`（橋/箸/端）、`kaeru`（帰る/変える/蛙）など、文脈で一意に決まらない場合
- **語境界の曖昧性**：`konoyouni` を「この様に / 子の用に」、`ashitahareru` を「明日晴れる / 足下走れる」など、複数の区切りが成立する場合
- **固有名詞・ファイル名・コマンド名・人名**：`siori` がファイル名なのか「栞」なのか、`tanaka` が `tanaka.py` なのか「田中さん」なのか等、識別子としての可能性があるもの。**勝手にファイル名を推測して触らない**
- **数字の単位・桁**：`niman` が「2万」か「2 man（マン）」か等
- **肯定/否定の判定が割れるとき**：`yamete` が「やめて（停止指示）」か「やってみて」のタイポか不明な場合は、**破壊的操作の前に必ず確認**

## 確認の仕方

短く、解釈候補を列挙して `AskUserQuestion` で聞く。例：

> 「`kamiwosutete` は次のどちらですか？
> 　1. 紙を捨てて（ファイル削除）
> 　2. 髪を切って（別の話題）」

## タイポ補正の範囲

- 明らかな typo（`nn` の重複、`tu`/`tsu` 揺れ、ローマ字綴り違い）は黙って補正してよい。
- ただし**補正が意味を変える可能性がある場合**（例：`kesite` → 「消して」か「決して」か）は確認する。

## やってはいけないこと

- ローマ字を「英語」として扱って解釈する（`kami` を camera 関連と誤読する等）。日本語ローマ字を最優先で疑う。
- 曖昧なまま破壊的なシェルコマンド・ファイル編集・git 操作を実行する。
